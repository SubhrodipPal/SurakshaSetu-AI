🎯 1. Intent Model

What is Intent in SurakshaSetu-AI?

Intent represents the structured, verifiable purpose behind every action the AI system wants to perform. No action can occur without a formally declared intent — and that intent must survive cryptographic verification before any tool is executed.

Every intent produced by OpenClaw is:

  • Deliberate — the result of multi-step AI reasoning, not ad-hoc commands
  • Traceable — carries a full reasoning chain explaining why the action is needed
  • Immutable — once signed, any modification invalidates it
  • Verifiable — ArmorClaw independently verifies the intent was created legitimately

Intent Token Generation

The intent token is generated using an HMAC-like hash that binds the plan contents to a shared secret:

  1. Serialize the plan's tool calls (tool names + parameters only)
  2. Concatenate with shared secret key
  3. Compute a deterministic hash of the combined string
  4. Produce token: setu-{hash}-{timestamp}
  5. Store hash inside the plan object for later integrity check
Critical Property: If anyone modifies the plan's tool calls after the token is generated, the hash will no longer match — and ArmorClaw will detect the tampering.

📜 2. Policy Model

Design Philosophy

The policy model is deterministic and static. There is no runtime policy relaxation, no dynamic rule adjustment, and no exception mechanism. The policy is hardcoded at deployment time and applies equally to every intent.

Design Decision: Dynamic policy modification is itself a security vulnerability. If an attacker compromises the reasoning layer, they cannot instruct the policy layer to relax its rules.

Policy Definition

{
  "allow": [
    "get_network_status",    // Read mesh state
    "isolate_node",          // Remove failing node from mesh
    "update_routing",        // Recalculate data paths
    "notify_admin",          // Send alert to human operator
    "reset_node"             // Restart a single node
  ],
  "deny": [
    "shutdown_all_nodes",    // Catastrophic action
    "reset_entire_network",  // Mass disruption
    "erase_logs",            // Destroys audit trail
    "format_storage",        // Data destruction
    "disable_auth",          // Security removal
    "modify_firmware"        // Unauthorized changes
  ],
  "maxToolCallsPerPlan": 10,
  "requireIntentToken": true,
  "requirePlanIntegrity": true
}

Evaluation Rules

Rule Priority Behavior
Tool in deny list Highest Immediately blocked, violation logged
Tool NOT in allow list High Blocked (fail-closed), treated as unknown
Tool IS in allow list Normal Permitted to proceed
Plan exceeds max tool calls High Entire plan blocked
No intent token present Highest Entire plan blocked
Plan integrity hash mismatch Highest Entire plan blocked, tampering logged

🛡️ 3. Enforcement Mechanism

5-Check Validation Pipeline

ArmorClaw implements a strictly ordered 5-check validation pipeline. Every check must pass for the plan to be approved. Any single failure causes the entire plan to be denied — this is the fail-closed principle.

1️⃣Intent Token Presence — Does the plan carry a cryptographic token? If no token is present, the entire plan is immediately denied.
2️⃣Token Issuer Validation — Was the token issued by OpenClaw-ReasoningAgent? Rejects tokens from unknown or spoofed issuers.
3️⃣Plan Integrity (Hash) — Re-computes the HMAC hash from plan contents and compares with the signed hash. Detects any post-signing modification (tampering).
4️⃣Per-Tool Policy Compliance — Each tool call is checked against the deny list first (deny wins), then the allow list. Unknown tools are blocked (fail-closed).
5️⃣Tool Call Limit — Ensures the plan does not exceed the maximum of 10 tool calls per plan, preventing flood attacks.

Hash Integrity Deep Dive

ArmorClaw independently recomputes the plan hash and compares it with the stored hash:

1. Extract tool names + parameters → serialize to JSON
2. Concatenate with shared secret key
3. Compute deterministic hash using bitwise operations
4. Compare computed hash with plan.hash
5. If mismatch → PLAN_TAMPERED violation → DENY entire plan
Defense in Depth: Even if an attacker could forge the hash (requiring the secret key), the policy check would still independently block denied tools. Both security layers must be defeated simultaneously.

Decision Audit Record

Every enforcement decision (approved or denied) produces a complete, append-only audit record containing: decision ID, timestamp, plan ID, all check results, final verdict, approved tool calls (or empty), and violation details.

Step 2 — OpenClaw Anomaly Detection

The AnomalyDetector runs threshold checks with rolling window trend analysis. Node B triggers: VOLTAGE_CRITICAL, PACKET_LOSS_CRITICAL, LINK_STABILITY_CRITICAL, VOLTAGE_DECLINING_FAST. Combined risk score: 0.95 (CRITICAL).

Step 3 — OpenClaw Failure Prediction

The FailurePredictor runs logistic regression on extracted features (voltage variance, packet drop trend, link instability, risk score). Result: failure probability = 78%, exceeds 65% threshold → shouldAct = TRUE.

Step 4 — OpenClaw Route Optimization

The RouteOptimizer (Modified AODV) finds alternative paths excluding Node B. Path scoring uses signal strength, stability, hop count, and latency. Best route to F: A → C → E → F (score: 0.807).

Step 5 — Plan Generated & Signed

Plan:
  1. isolate_node({ nodeId: "B" })       ← in allow list ✓
  2. update_routing({ nodeId: "B" })     ← in allow list ✓
  3. notify_admin({ message: "..." })    ← in allow list ✓

Intent Token: "setu-a3f8b21c-m2k9x"

Step 6 — ArmorClaw Validates

✅Intent Token Presence — Token: setu-a3f8b21c-m2k9x
✅Token Issuer — Verified: OpenClaw-ReasoningAgent
✅Plan Integrity — Hash verified: a3f8b21c matches
✅Policy — isolate_node, update_routing, notify_admin all in allow list

Executor sends approved calls to Gateway: Node B is isolated from the mesh, routing tables are rebuilt, and an admin notification is dispatched. The mesh self-heals automatically.

Why this was allowed: Every tool call is in the allow list, the intent token is valid, the plan hash is intact, and the actions are targeted (single node) and reversible. The 4-step reasoning chain provides a complete audit trail.

🛑 5. Explanation of Blocked Action

Scenario A: Unauthorized Command — shutdown_all_nodes()

An external request attempts to shut down the entire IoT network — a catastrophic action that ArmorClaw must block.

The Unauthorized Plan

Plan:
  1. shutdown_all_nodes({})    ← IN DENY LIST!

Intent Token: "setu-f7e2a91b-n3p8r"  (valid format, valid signature)
Hash: "f7e2a91b"                      (hash is correct)
Note: The plan has a valid token and valid hash — the format is correct. But the content violates policy.

ArmorClaw Validation

✅Intent Token Presence — Token present
✅Token Issuer — Issuer matches
✅Plan Integrity — Hash matches (plan was not tampered)
❌Policy: shutdown_all_nodes — DENIED — Tool is in the deny list
✅Tool Call Limit — 1 ≤ 10
🛑 DENIED — Violation: DENIED_TOOL — "shutdown_all_nodes" is explicitly denied by policy

Result: No tool calls executed. Gateway never receives any command. Network remains 100% operational and unaffected. Violation logged in enforcement audit trail.

An attacker intercepts a legitimate plan and injects erase_logs after the intent token was already generated.

The Tampered Plan

Original plan (signed with hash "b4c9d3e1"):
  1. isolate_node({ nodeId: "B" })
  2. notify_admin({ message: "..." })

↓↓↓ INJECTED AFTER SIGNING ↓↓↓
  3. erase_logs({})                    ← TAMPERED!

ArmorClaw Validation

✅Intent Token Presence — Token present
✅Token Issuer — Issuer matches
❌Plan Integrity — Hash mismatch! Expected: b4c9d3e1, Computed: 7a1f5c82 — PLAN TAMPERED
✅Policy: isolate_node — ALLOWED
✅Policy: notify_admin — ALLOWED
❌Policy: erase_logs — DENIED — Tool is in the deny list
✅Tool Call Limit — 3 ≤ 10
🛑 DENIED — 2 Violations: PLAN_TAMPERED + DENIED_TOOL
Defense in Depth: Two independent security layers caught this attack. The integrity check detected modification (hash mismatch). The policy check independently caught erase_logs as denied. Both layers must be defeated simultaneously — and neither can be bypassed.

🔒 6. Separation Between Reasoning and Execution

The Cardinal Rule

Fundamental Security Principle

The reasoning layer (OpenClaw) can THINK but cannot ACT.
The execution layer (Executor) can ACT but cannot THINK.
ArmorClaw sits between them, ensuring every thought is VALIDATED before becoming an action.

How Separation is Enforced in Code

OpenClawAgent.js — imports ONLY reasoning modules (AnomalyDetector, FailurePredictor, RouteOptimizer). It has zero access to Gateway or Executor. It cannot call any tool.

Executor.js — imports ONLY the Gateway reference. It has zero access to OpenClaw. It cannot generate plans or reason about telemetry.

SystemOrchestrator.js — the ONLY place where layers connect, enforcing the strict sequential pipeline: OpenClaw.reason() → ArmorClaw.validate() → Executor.execute(). There is no path from reasoning to execution that skips ArmorClaw.

Why This Matters

  1. Compromised AI cannot act: If the reasoning model is manipulated, it can only produce plans — it cannot execute them. Plans still must pass ArmorClaw's deterministic checks.
  2. Compromised executor cannot think: If the execution layer is compromised, it cannot generate its own actions. It can only run what ArmorClaw explicitly approves.
  3. ArmorClaw is deterministic: The enforcement layer uses no AI, no probabilistic reasoning, and no learned behavior. It evaluates static rules against structured data. It cannot be confused or tricked through prompt injection.
  4. No hidden execution paths: The SystemOrchestrator enforces a strictly sequential pipeline with no alternative paths, callbacks, or event handlers that bypass ArmorClaw.
  5. Observable and auditable: Every plan, decision, and execution is logged and rendered in the dashboard UI. Nothing happens silently.
